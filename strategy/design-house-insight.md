## 一、什么是方案设计公司(Design House)

### 1.1 定义与产业链位置

方案设计公司（Design House）是产业链中的**智力密集型环节**——不拥有工厂、不持有品牌终端产品，核心交付物是**设计方案 + SDK/固件 + 技术支持**。客户拿到方案后自行量产或委托代工。

Design House的本质是**将技术能力打包为可复用的解决方案**。它是芯片厂商和终端品牌之间的**翻译层**——把芯片的潜力转化为客户可直接落地的产品方案。

### 1.3 Design House的核心特征

**四大特征**拆解：

| 特征 | 内涵 | 对QLB的意义 |
|------|------|------------|
| **轻资产** | 无工厂、无产线、无大量库存 | 成都初创，无需深圳制造基础设施 |
| **设计驱动** | 核心价值在设计能力，非制造能力 | Co-Agent架构+PDP协议=设计IP |
| **SDK生态** | 通过SDK锁定客户，降低迁移成本 | Py4OH/Co-Agent SDK已有基础 |
| **外包制造** | 制造环节外包给代工厂 | 深圳/成都代工资源丰富 |

**轻资产结构的财务优势：**

| 指标 | Design House | 产品公司 | 制造企业 |
|------|-------------|---------|---------|
| 固定资产占比 | <5% | 15-30% | 40-60% |
| 研发人员占比 | 60-80% | 30-50% | 10-20% |
| 盈亏平衡营收 | ¥300-500万 | ¥1,000-3,000万 | ¥5,000万+ |
| 现金流周期 | 30-60天 | 90-180天 | 120-240天 |
| 扩张边际成本 | 极低（复用方案） | 中（新产品研发） | 高（新产线） |

### 1.4 产品公司的天花板——M5Stack被收购案例

M5Stack是"Design House vs 产品公司"路径抉择的**完美反面教材**。

**M5Stack的成功与困局：**

| 阶段 | 时间 | 事件 | 估值/规模 |
|------|------|------|----------|
| 创立 | 2017 | Jimmy Lai创立，HAX加速 | 种子期 |
| 产品爆发 | 2018-2022 | **615+** SKU，日本**~35%**市场份额 | 快速增长 |
| 社区繁荣 | 2019-2023 | **5,856** Hackster成员 | 开发者生态 |
| 乐鑫天使投资 | 2018 | 乐鑫少数股权投资 | 芯片厂入局 |
| **被收购** | **2024.06** | **乐鑫获得~57%控股权** | 未达披露标准 |

**被收购的结构性原因：**

```
┌──────────────────────────────────────────────────────┐
│           M5Stack的"产品公司陷阱"                      │
│                                                      │
│  ┌─────────────────┐                                 │
│  │ 核心依赖：ESP32   │ ← 单一芯片供应商                │
│  └────────┬────────┘                                 │
│           │                                          │
│  ┌────────▼────────┐                                 │
│  │ 产品公司路径      │                                 │
│  │ · 615+ SKU       │ ← 库存管理压力                  │
│  │ · 自有品牌        │                                 │
│  │ · 硬件毛利30-40%  │ ← 低于Design House             │
│  └────────┬────────┘                                 │
│           │                                          │
│  ┌────────▼────────────────────────────────┐         │
│  │ 结构性困局：                              │         │
│  │ 1. 芯片方：乐鑫掌握上游话语权             │         │
│  │ 2. SKU膨胀：管理成本随产品线性增长         │         │
│  │ 3. 品牌溢价有限：ESP32不可替代=被锁定     │         │
│  │ 4. 融资困难：硬件产品公司估值天花板低       │         │
│  └────────┬────────────────────────────────┘         │
│           │                                          │
│  ┌────────▼────────┐                                 │
│  │ 最终结果：         │                                │
│  │ 被芯片供应商收购   │ ← 从独立公司变为乐鑫子公司      │
│  │ 收购价未达披露标准  │ ← 暗示估值不高                 │
│  └─────────────────┘                                 │
└──────────────────────────────────────────────────────┘
```

**Design House路径的反事实假设：** 如果M5Stack选择Design House路径（卖方案而非卖产品），它可以：
- 服务多个芯片平台，不被单一供应商锁定
- 方案复用带来更高毛利（50-75% vs 30-40%）
- 轻资产结构下现金流更健康
- 对芯片厂的议价权更强（"我可以换芯片"）

### 1.5 洞察小结

> **核心洞察**：Design House是AI硬件创业的最优起步模型——轻资产、高毛利、设计驱动、可复用。产品公司看似"有品牌"，实则容易陷入SKU膨胀、库存压力、供应商锁定的三重陷阱。M5Stack被乐鑫收购是最新的反面案例。

**对你的启示：**

**可以直接照做：**
- [ ] 明确自身定位为Design House，不追求成为产品公司
- [ ] 核心交付物定义：方案设计+Co-Agent SDK+PDP协议+BOM+测试规范
- [ ] 制造环节100%外包（深圳/成都代工）

**需要调整：**
- [ ] 现有"AI Agent硬件平台公司"定位需升级为"AI Agent Design House"
- [ ] 熊猫派3.0等自有产品线逐步转为Design House的参考设计/Demo，而非核心业务

### 2.4 洞察小结

> **核心洞察**：全球成功的Design House有两个共同点——统一SDK平台和开发者生态。Nordic以51.8%毛利率和$26.9亿市值证明了Fabless DH的上限；Particle以$20M ARR和2.5x ARR收购倍数证明了平台型DH的估值逻辑。M5Stack的低价被收购则警示：没有平台化能力的硬件公司，终将被上游吃掉。

**对你的启示：**

**可以直接照做：**
- [ ] 以Nordic的SDK平台化策略为参考，打造Co-Agent统一SDK
- [ ] 学习Particle的ARR模型——硬件一次性收入 + 云服务/SDK授权订阅
- [ ] 保持芯片无关性：Co-Agent支持ESP32/RK3588/Hi3861多芯片

**需要调整：**
- [ ] 不走M5Stack的SKU扩张路线，而是以方案复用为核心增长逻辑
- [ ] 远期考虑A股/港股上市路径（Nordic/乐鑫模式），而非被收购退出

## 八、Design House的风险图谱

### 8.1 轻资产结构性风险

| 风险类型 | 描述 | 严重度 | 概率 | QLB暴露度 |
|---------|------|--------|------|----------|
| **品控风险** | 制造外包导致品质不可控 | 高 | 中 | ⭐⭐⭐ |
| **交付风险** | 代工厂产能/排期不可控 | 中 | 中 | ⭐⭐⭐ |
| **IP风险** | 方案被代工厂/客户抄袭 | 高 | 中高 | ⭐⭐⭐⭐ |
| **供应链风险** | 关键芯片缺货/涨价 | 高 | 中 | ⭐⭐⭐ |

**IP保护策略：**

| 层级 | 保护方式 | 效果 |
|------|---------|------|
| 法律层 | 专利+商业秘密协议 | 基础保护 |
| 技术层 | SDK加密+License验证 | 中等保护 |
| 生态层 | **SDK持续迭代 > 方案文件本身** | **最有效** |
| 商业层 | 客户粘性（切换成本高） | 长期保护 |

> **关键洞察**：Design House的IP保护不能依赖法律手段——方案文件一旦交付就可能被复制。真正的护城河是**SDK的持续迭代能力**和**客户切换成本**。

### 8.2 客户相关风险

| 风险类型 | 描述 | 严重度 | 概率 | 应对 |
|---------|------|--------|------|------|
| **客户集中度** | 单一大客户占比过高 | 高 | 中 | 分散客户组合 |
| **被替代风险** | 客户自建团队取代DH | 中 | 中高 | 持续创新+SDK锁定 |
| **被收购风险** | 被芯片厂或大客户收购 | 中 | 低中 | 保持多芯片+多客户 |
| **回款风险** | 政府/教育项目回款慢 | 中 | 高 | 预付款+分期 |

**被替代风险深度分析：**

| 客户自建条件 | 应对策略 |
|-------------|---------|
| 客户有足够研发团队 | 让SDK复杂度超过客户自建成本 |
| 客户只需要一次性方案 | 设计为需要持续更新的订阅模式 |
| 芯片厂提供免费参考设计 | 提供芯片厂不愿做的应用层方案 |
| 开源方案足够好 | 在开源基础上提供商业级支持+定制 |

### 9.2 Co-Agent + PDP协议核心IP价值

| IP资产 | 描述 | 竞争壁垒 | DH价值 |
|--------|------|---------|--------|
| **Co-Agent三层架构** | LLM Cloud + Brain Core + Peripheral Nodes | 高（架构设计难复制） | 方案核心交付物 |
| **PDP v1协议** | 外设通过UART自声明能力，Brain Core自动注册 | 高（先发+标准化） | 客户集成接口规范 |
| **工具注册机制** | LLM自然语言驱动硬件 | 中高 | AI Agent核心能力 |
| **多芯片抽象层** | ESP32→RK3588→Hi3861统一API | 高 | 芯片无关性保障 |
| **18,000行设计文档** | 中英日三语，完整架构文档 | 中（可复制但有先发） | DH专业度证明 |

**PDP协议 vs M5Stack M-Bus对比：**

| 维度 | PDP v1协议 | M5Stack M-Bus |
|------|-----------|---------------|
| 定位 | AI Agent外设通信协议 | 物理总线接口标准 |
| 层级 | 应用协议层 | 硬件物理层 |
| 连接方式 | UART/I2C/SPI（软件定义） | 固定排针（硬件定义） |
| 扩展性 | 无限（任何UART设备） | 有限（需M5兼容外设） |
| AI集成 | ✅ 原生（工具自动注册→LLM调用） | ❌ 无（需额外开发） |
| 芯片锁定 | ❌ 无（芯片无关） | ✅ ESP32锁定 |
| 生态开放度 | 开放协议 | 半封闭生态 |
| DH适配性 | ⭐⭐⭐⭐⭐（方案级可复用） | ⭐⭐（产品级，难复用） |

### 9.6 洞察小结

> **核心洞察**：QLB的现有资产（Co-Agent+PDP+Py4OH+蜀鸿会+唐佐林网络）与Design House模式高度适配（8.0/10）。核心IP（Co-Agent三层架构+PDP协议）在OH生态中具有独特性，竞争对手尚无类似方案。最大挑战是从"产品思维"转向"方案思维"——熊猫派3.0从独立产品变为Design House的参考设计。

**对你的启示：**

**可以直接照做：**
- [ ] Co-Agent SDK重构为"客户集成友好"版本（标准API+文档+示例）
- [ ] PDP协议发布为开放标准（类似M-Bus但更高层级）
- [ ] 唐佐林的OH MVP/HDE身份转化为DH业务背书

**需要调整：**
- [ ] 团队补充BD/方案销售角色（技术型销售）
- [ ] 思维转换：每个项目的终极目标不是"交付一个产品"，而是"沉淀一套可复用方案"
